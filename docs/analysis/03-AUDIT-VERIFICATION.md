## B. Confirmed open issues (ranked)

Severity recalibrated by impact-on-common-path. `★` = re-confirmed by direct code read in this session.

### HIGH

- **NEW-screenshot-1 ★** — `view_screenshot` / `drawing_export_pdf` build a `matplotlib.pyplot`
  figure **inside an `asyncio.to_thread` worker without forcing the `Agg` backend**
  (`ezdxf_backend.py:1420-1444`). On a host whose default MPL backend is GUI (Qt/Tk) this can
  **hard-crash the process (SIGSEGV)** — this is the real cause of the `pytest` warning
  "Starting a Matplotlib GUI outside of the main thread will likely fail." `drawing_finalize`
  calls `view_screenshot`, so finalize can take down the server.
  **Fix:** render via `matplotlib.figure.Figure` + `FigureCanvasAgg` (no `pyplot` global state),
  or `matplotlib.use("Agg")` before the pyplot import. Headless-testable.

- **I15 ★** — `drawing_finalize` (`server.py:2333`) runs **only** `DrawingValidator().run()` and
  **never** `backend.drawing_critique` / `run_critique`. The 6 premium critique focuses
  (iso128, layer_color, untrimmed_corner, duplicate_entities, construction_left, dim_overlap)
  are **not part of the finalize gate** — a drawing passes finalize with untrimmed corners,
  non-ISO-128 lineweights, leftover construction scaffolding, duplicate lines. The product's
  entire differentiator ("critique-then-finalize", CLAUDE.md) is advisory, not enforced.
  **Fix:** run `run_critique(focus=None)` inside `drawing_finalize`, merge into the report,
  gate (or warn) on critique errors. This is the moat.

- **NEW-AUTH-1 ★** — `_validate_http_bind(host)` is called **only** in the
  `if __name__ == "__main__"` block (`server.py:2940`). The documented launch command
  `fastmcp run server.py:mcp --transport http --host 0.0.0.0` *imports* the module and never
  runs `__main__`, so the remote-bind guards (refuse remote bind without `ALLOW_REMOTE_HTTP`,
  refuse without `MCP_AUTH_TOKEN`) **never fire** under the recommended launch. With the token
  unset, the server binds to `0.0.0.0` anonymously with nothing to stop it.
  **Fix:** enforce the bind check at import time / in the FastMCP lifespan, not only in `__main__`.

- **NEW-gear-roottooth + R21 ★** — `generate_involute_flank` (`gear.py:40-50`) prepends
  `(root_r, 0)` while the involute starts at `(base_r, 0)` — both on y=0, so for `root_r < base_r`
  the root→base join is a **straight radial segment, not a trochoid** (R21). Worse, for
  `root_r ≥ base_r` (teeth ≥ ~42 @20° PA, within the teeth≥6 API limit) no root point is added and
  **the flank starts at `base_r` *inside* the root circle** → the tooth profile dips below the root
  arc and the closed outline **self-overlaps** (z=50 reproduced: outline points inside root_r).
  **Fix:** clamp the flank start to `max(base_r, root_r)`; skip/relocate the root arc when
  `root_r ≥ base_r`; ideally a true trochoid below the base circle.

### MED

- **R4 ★** — `dim_overlap` critique is a **guaranteed no-op**: `_check_dim_overlap`
  (`critique.py:199-201`) reads `text_position`/`defpoint`/`insertion_point` which **no backend
  populates** for dimensions; it also filters `entity_list(layer_filter="DIM")` (`critique.py:192`)
  which misses the iso13567 dim layer. **Fix:** populate a dim reference point in both extractors
  (or fall back to bbox center), and resolve the dim layer from the active set.

- **N4** — `dimension_auto` hardcodes `layer="DIM"` (`base.py:948/962/970/977`); construction ops
  hardcode `"CONSTRUCTION"` (`base.py:869/880`); `_check_construction_left` also hardcodes
  `"CONSTRUCTION"` (`critique.py:99`). For the **iso13567** set (dim layer `M-DIMEN-T-N`,
  construction `M-CONST-E-N`) dims land on the wrong layer and leftover scaffold is never caught.
  **Fix:** resolve dim/construction layer from the active `LAYER_SET_REGISTRY` keyed on `PlanSpec`.

- **R13-COM / NEW-offset-1 ★** — `ComBackend.entity_offset` (`com_backend.py:860-874`) accepts
  `side_x/side_y` but never reads them (`ent.Offset(distance)` only) → COM offsets to a
  backend-decided side while ezdxf honors the hint. Same tool, opposite results.
  Also: only `offset_ent[0]` is kept; extra entities `Offset` returns leak. **Fix:** honor side in COM.

- **N2 (tool)** — standalone `drawing_save_as` MCP tool defaults `fmt='dwg'` and doesn't derive from
  the extension, so `drawing_save_as('part.dxf')` raises on ezdxf / writes DWG bytes into a `.dxf`
  name on COM. **Fix:** derive fmt from the path extension in the tool too (mirror finalize).

- **R5** — COM executor calls `CoInitialize()` but **`CoUninitialize` appears nowhere**; on every
  `SendCommand` timeout the stuck STA thread + its COM apartment leak (a fresh executor is built),
  and `disconnect()` uses the same `wait=False`. Long live-AutoCAD sessions exhaust resources.
  **Fix:** worker-side `try/finally` `CoUninitialize`; bound abandoned executors.

- **R16** — COM `transaction_begin/commit/rollback` set `_transaction_active` *after* the await
  with no `try/finally` (`com_backend.py:1398/1407/1417`) → stale flag on error; rollback's raw
  `_UNDO B` (`:1414`) bypasses the CMDACTIVE guard. **Fix:** `try/finally`; route via `_safe_send_command`.

- **R23 / N6** — `system_run_lisp` (`com_backend.py:1483-1488`) sends raw LISP with **no CMDACTIVE
  guard** (deadlock on a prompting form) and captures the **void** `SendCommand` return so it
  **always reports `result:"nil"`**. **Fix:** route through `_safe_send_command`; capture via
  `(setvar "USERS1" …)` + `GetVariable`, or drop the bogus `result`.

- **N5** — `EntityInfo.properties` shape diverges: ezdxf never sets `bounding_box` (COM always does);
  ezdxf TEXT sets `rotation` / MTEXT `char_height`, COM sets neither. Clients break on one backend.
  **Fix:** pick one contract; enforce both ways.

- **N3** — `entity_select_smart` `length_range` rejects **all ARCs** (no extractor sets ARC `length`).
  **Fix:** populate ARC `length = radius·|Δangle|` or compute on the fly.

- **NEW-dimauto-baseline-rotation** — `dimension_auto` baseline mode (`base.py:952-963`) offsets
  perpendicular to a rotated baseline but omits `rotation=` → dim lines render horizontal.
- **NEW-dimauto-ordinate-misnomer** — `dimension_auto` "ordinate" (`base.py:965-979`) emits two
  linear dims at a constant offset → overlap; not real ordinate dims.

- **NEW-com-chamfer-return** — `entity_chamfer` (`com_backend.py:1609-1629`) hunts for a new
  `AcDbLine` that CHAMFER of two distinct lines never creates → almost always returns the wrong entity.
- **NEW-com-set-variable-coercion** — `system_set_variable` (`com_backend.py:1453-1458`) forwards the
  raw value to `SetVariable` with no int/float coercion → type mismatch for numeric sysvars.

- **NEW-section-dupbore** — `gear_draw_section_aa` + keyway draws the two bore lines twice →
  `duplicate_entities` critique fires and blocks the premium workflow.
- **NEW-validator-keyway** — validator step-4 keyway heuristic requires a closed polyline, but
  `draw_keyed_bore` emits `closed=False` and the section keyway is 4 lines → permanent
  false-negative (`keyway_section_unverified`) even when a keyway exists.
- **NEW-gear-baseconstr** — the gear base circle is placed on `CONSTRUCTION` and is silently deleted
  by the **mandated** `construction_clear()` before finalize → the returned handle dangles.

- **R15** — `system_about` `tool_groups` static dict omits ~22 registered tools (all engineering +
  premium + corner-ops + `drawing_close`) and misfiles `entity_delete_many`. **Fix:** derive groups
  from tags dynamically.

- **NEW-test-com-1** — `normalize_lineweight` table never tests the ambiguous bare ints `1`/`2`
  (`normalize_lineweight(2)==200`, i.e. 2.00 mm not 2 hundredths) — the exact P0 truncation hazard.
- **NEW-test-com-2** — COM `transaction_begin/commit/rollback` + the `system_run_command` CMDACTIVE
  busy-guard are entirely untested (R6 residual).

- **NEW-offset-attrs (ezdxf)** — `entity_offset` copies only `layer`, dropping source color/linetype,
  so offsetting a CENTER/colored line yields a Continuous ByLayer line (unlike copy/mirror).
